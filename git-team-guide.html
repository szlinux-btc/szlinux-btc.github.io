一、为什么需要分支模型？

一个人用 Git 可以很随意，但一个 3 人以上的团队如果没有统一的分支策略，就会陷入混乱：代码互相覆盖、线上 Bug 找不到负责人、想回滚却不知道哪个版本安全。

分支模型本质上是一套 团队协作的"交通规则"：规定谁能在哪里开车（写代码）、什么时候可以并道（合并）、紧急情况怎么处理（热修复）。没有规则，交通就会瘫痪。

核心矛盾：并行开发的需求 vs 代码稳定性的要求。多人同时在 main 分支上提交，等于没有分支。好的分支模型让并行开发成为可能，同时保证主线始终可发布。

二、GitFlow：最经典的分支策略

GitFlow 由 Vincent Driessen 于 2010 年提出，至今仍是中大型项目的主流选择。它定义了 5 种长期分支和若干临时分支。

2.1 五种核心分支

分支 用途 合并来源 谁可以动
main 生产环境代码，只放稳定可发布的版本 release / hotfix Release Manager
develop 集成了所有已完成的 feature，作为下一版本的开发基准 feature / release 所有开发者
feature/* 具体功能开发，每个功能一个分支 从 develop 创建，最终合并回 develop 功能负责人
release/* 发布前的最后打磨，只修 Bug，不加新功能 从 develop 创建，合并到 main + develop 测试团队
hotfix/* 线上紧急修复，从 main 拉出，修复后合并回 main + develop 从 main 创建，合并到 main + develop 运维 / 值班人员

2.2 分支流向图

━━━━ main ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ ↖ ↗ ↖ ↗ [hotfix] [hotfix] ↖ ↗ ↖ ↗ ━━━━ release/1.0 ━━━ ━━ release/1.1 ━━▶ ↗ ↘ ↗ ↘ ━━━━ develop ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ ↗ ↘ ━ feature/login feature/pay ━

2.3 日常操作示例

开始一个新功能：

git checkout develop git pull origin develop git checkout -b feature/user-auth # 开发中... git add . git commit -m "feat: 添加用户认证模块" git push -u origin feature/user-auth

功能完成，合并到 develop：

git checkout develop git pull origin develop git merge --no-ff feature/user-auth git push origin develop # 删除已合并的功能分支 git branch -d feature/user-auth

紧急热修复线上 Bug：

git checkout main git pull origin main git checkout -b hotfix/payment-crash origin/main # 修复 Bug... git commit -m "hotfix: 修复支付页面崩溃问题" git checkout main git merge --no-ff hotfix/payment-crash git tag -a v1.2.1 -m "发布 1.2.1 紧急修复版本" git push origin main --tags git checkout develop git merge --no-ff hotfix/payment-crash git push origin develop

💡 --no-ff 的作用：禁用 fast-forward 合并，Git 会创建一个新的合并提交，即使可以 fast-forward 也如此。这样在历史记录中能清晰看到每个功能分支的开始和结束，方便回溯。

三、Trunk-Based：快节奏团队的选项

Trunk-Based Development（TBD）是一种更激进的策略：所有开发者都在同一个分支（trunk/main）上工作，feature 通过短生命周期的特性开关（Feature Toggle）控制上线与否。

━━━━ main/trunk ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ ━── feature/login ──▶ ━─ feature/pay ──▶ ── feature/dark ─▶ ── feature/vip ──▶

⚠️ TBD 的前提条件：团队必须有成熟的 CI/CD 流水线、自动化测试覆盖率 > 80%、特性开关框架（LaunchDarkly / Unleash）。没有这些基础设施，TBD 会把团队拖入灾难。

TBD 典型流程：

# 1. 每天开始，先同步主线 git checkout main git pull --rebase origin main # 2. 在特性开关内开发 git checkout -b feature/new-checkout # 代码中使用特性开关: # if (featureToggle.isEnabled('new-checkout')) { ... } git commit -m "feat: 新结账流程（特性开关控制）" git push origin feature/new-checkout # 3. 合入主线（PR 后自动合并，不等特性上线） # PR 合并后，代码已在 main，但功能通过开关控制 # 4. 功能验证完毕，开启开关 # 特性开关平台 → 开启 new-checkout → 灰度放量

💡 GitHub Flow：如果你觉得 GitFlow 太重但又想要分支保护，可以考虑 GitHub Flow——只有 main + feature 分支，PR 审查合并，更轻量，适合持续部署的小团队。

四、两种模型横向对比

维度 GitFlow Trunk-Based
分支数量 5 种长期分支 1 个主干 + 短期特性分支
适用团队规模 中大型（10 人以上） 小型（10 人以下）
发布频率 周期发布（2~4 周一个版本） 持续发布（每天甚至每小时）
多版本并行维护 ✅ 支持 release 分支 ❌ 通常只维护一个版本
Hotfix 处理 ✅ 独立的 hotfix 分支 ⚠️ 需切换或 cherry-pick
学习成本 较高，规则复杂 较低，流程简单
CI/CD 要求 中等 非常高（自动化测试必须到位）
适合场景 需要多版本并行、有明确发布周期的项目 追求快速迭代、CI/CD 成熟的 SaaS 产品

💡 实用建议：大多数中小团队（3~8人）不需要 GitFlow 的全部复杂性。用 简化版 GitHub Flow 即可：main 分支受保护 → feature 分支 PR 审查 → 合并后自动部署。复杂度和收益最均衡。

五、冲突解决的完整流程

冲突的本质是：两个分支修改了同一文件的同一处，Git 不知道该保留哪个。对于运维场景，常见冲突来源是配置文件（nginx.conf、.env、数据库连接配置）。

5.1 冲突是如何产生的

# 场景：Alice 和 Bob 同时修改了 config.py # Alice 先提交了 git commit -m "feat: 更新数据库连接" git push origin feature/payment # Bob 也提交了（此时远程已有 Alice 的改动） git commit -m "fix: 修复超时问题" git push origin feature/payment ! [rejected] feature/payment -> feature/payment (fetch first) # Git 拒绝推送：远程有你的本地不知道的提交

5.2 正确的解决步骤

Step 1 — 拉取远程最新代码（不要直接强推）：

git fetch origin git merge origin/feature/payment # 或者用 rebase（更推荐，保持线性历史） git rebase origin/feature/payment

Step 2 — Git 报告冲突，打开文件查看：

CONFLICT (content): Merge conflict in config.py git status # 冲突文件会显示为 unmerged 状态

Step 3 — 手动解决冲突文件：

<<<<<<< HEAD (当前分支) host = "prod-db.internal" port = 3306 ======= host = "10.0.1.100" port = 5432 >>>>>>> origin/feature/payment (远程分支)

按实际需求保留正确的版本，或合并两者：

# 保留两个改动中最合理的结果 host = "prod-db.internal" port = 3306 # 删除 Git 的冲突标记 <<<<<<< ======= >>>>>>>

Step 4 — 标记冲突已解决，提交：

git add config.py git commit -m "merge: 解决 config.py 冲突" git push origin feature/payment

5.3 用工具更高效地解决冲突

命令行
git mergetool
调用系统配置的合并工具（Beyond Compare / Meld / VSCode）三栏对比解决
VSCode
内置冲突编辑器
点击"接受当前更改/接受传入更改/保留双方"，可视化解决
IDE
JetBrains 全家桶
强大的图形化合并界面，支持批量解决相同冲突
GitHub
网页端编辑
PR 界面可直接在浏览器解决轻量冲突

⚠️ 解决冲突的铁律：

  • 永远不要在解决冲突时直接删掉别人的改动
  • 不确定时，找改动者当面沟通
  • 解决完必须运行测试，确认功能正常
  • 禁止使用 --ours 或 --theirs 无脑覆盖

六、团队协作最佳实践

6.1 Commit Message 规范

清晰的提交信息是代码考古学的基础。推荐使用 Conventional Commits 规范：

type(scope): subject # type 类型： feat 新功能 fix Bug 修复 docs 文档变更 style 格式调整（不影响功能） refactor 重构（不修复问题不加功能） perf 性能优化 test 测试相关 chore 构建/工具/依赖更新 # 示例： feat(auth): 添加 OAuth2 第三方登录 fix(payment): 修复微信支付回调超时问题 refactor(db): 重构用户表查询逻辑 chore: 升级 Node.js 到 22.x

6.2 PR 审查 Checklist

每次 Pull Request 合入前，确保以下事项已确认：

  • PR 描述清晰说明了改了什么、为什么改、怎么测试
  • 代码通过了所有 CI 自动化测试（单元测试、集成测试）
  • 没有敏感信息泄露（密码、密钥、Token 硬编码）
  • 复杂逻辑有注释，公共 API 有文档
  • 相关配置文件同步更新（如有必要）
  • 至少 1 人 review 通过（非本人）
  • 分支已 rebase 到最新 main/develop，无冲突
  • 合并方式使用 --no-ff 或 Squash Merge（保持主线整洁）

6.3 保护关键分支

在 GitHub / GitLab 上给 main 和 develop 设置分支保护规则：

强制设置
Require pull request reviews
合并前必须有至少 1 个 approved review
强制设置
Require status checks to pass
CI 测试通过后才能合并
强制设置
Dismiss stale reviews
新提交自动清除旧的 review 状态
推荐设置
Require branches to be up to date
合入前强制 rebase 到最新，避免埋下隐藏冲突

⚠️ 禁止：禁止任何人强制推送（force push）到 main/develop 分支，禁止任何人直接 commit 到受保护分支。如果需要紧急修改，必须走 PR 流程——这是团队代码安全性的底线。

七、工具链推荐

环节 推荐工具 用途
托管平台 GitHub / GitLab / Gitea 代码托管、PR/MR、CI 集成
代码审查 GitHub PR / GitLab MR / Phabricator 行级评论、建议修改、审批流
可视化合并 VSCode / Beyond Compare / Meld 图形化解决 Git 冲突
CI/CD GitHub Actions / GitLab CI / Jenkins 自动化测试、构建、部署
特性开关 Unleash / LaunchDarkly / 功能开关 Trunk-Based 必备，上线灰度控制
Commit 规范 commitlint + husky 强制执行 Conventional Commits 格式
图形化客户端 Sourcetree / GitKraken / GitHub Desktop 新手友好，分支可视化

总结：GitFlow 和 Trunk-Based 没有绝对的好坏，只有适合与否。选对分支模型，配置好保护规则，建立 Commit 规范——这三件事做好，一个团队的协作效率可以提升一倍以上。工具只是手段，流程才是核心。
